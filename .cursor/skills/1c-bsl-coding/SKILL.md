# Разработка кода на 1С (BSL)

Используй этот skill при генерации или правке кода 1С:Предприятие 8.3 (BSL) в проекте.

## Стиль кода

- **Именование:** PascalCase для переменных, процедур, функций, параметров (ТекущийПользователь, РассчитатьСумму). Константы — ВЕРХНИЙ_РЕГИСТР.
- **Ключевые слова** — стандартный регистр 1С: `Не`, `Или`, `И`, `Тогда`, `КонецЕсли`, `КонецЦикла`, `КонецПопытки`.
- **Отступы:** 4 пробела. Длина строки до 140 символов; длинные запросы переносить.
- **Символы:** в коде 1С не использовать HTML-сущности (`&amp;`, `&lt;`). Параметры запроса: `&Параметр`.

## Запросы

- Всегда параметры: `Запрос.УстановитьПараметр("Имя", Значение)`. Не конкатенировать строки в текст запроса (риск SQL-инъекции).
- Временные таблицы: `ПОМЕСТИТЬ` перед `ИЗ`. Для JOIN таблиц значений использовать `ВЫРАЗИТЬ` при необходимости приведения типов.

## Типичные ошибки (избегать)

- **Массив:** `Новый Массив(1, 2, 3)` создаёт многомерный массив. Нужен список — создать пустой массив и вызывать `Добавить()`.
- **Дата:** `Дата(Год, Месяц, День)` — порядок: год, месяц, день.
- **Проверка на пустоту:** использовать `ЗначениеЗаполнено()`, `ТипЗнч() = Тип("Неопределено")`, а не прямое сравнение с `Неопределено`.
- **Циклы/условия:** синтаксис 1С: `Пока ... Цикл ... КонецЦикла`, не `while`/`for` из других языков.

## Критическое мышление при анализе

- Перед ответом проверяй, нет ли у задачи нескольких корректных трактовок. Если трактовок несколько, явно перечисляй варианты и результат для каждого.
- Анализируй намерение автора отдельно от текста: что пользователь хотел получить, что код сделает фактически, есть ли расхождение.
- Для каждого решения оценивай риски: безопасность, производительность, поддерживаемость, совместимость с текущей конфигурацией.
- Не делай молчаливых допущений в неоднозначных местах. Либо зафиксируй выбранное допущение явно, либо задай уточняющий вопрос.
- При ревью и исправлениях сначала укажи потенциальные проблемы и причины, затем предложи минимально рискованное исправление.
- Если есть риск скрытой ошибки (например, неверный тип, неоднозначный конструктор, опасная конкатенация запроса), обязательно объясни, почему это риск.

## Архитектура и модули

- **Управляемые формы:** чётко разделять `&НаКлиенте` (интерфейс) и `&НаСервере` (запросы, бизнес-логика). Минимизировать количество серверных вызовов.
- **Попытка/Исключение:** использовать только для непредсказуемых ситуаций (внешние вызовы, файлы, парсинг). Не использовать для проверки прав, валидации и проверки заполненности — там явные проверки (`Если Не ЗначениеЗаполнено(...)` и т.д.).
- **Константы и глобальные переменные:** в проекте не использовать; передавать через структуру Параметров.

## Документирование

- Для экспортных и сложных процедур/функций — документирующий комментарий: Описание, Параметры, Возвращаемое значение (для функций), Пример. Комментарии на русском.
- Сложную логику и временные решения помечать TODO/FIXME с пояснением.

## Стандарты и инструменты

При генерации и правке кода учитывай стандарты и диагностики из **STANDARDS-AND-TOOLS.md** в этой папке skill:

- **v8-code-style** (1С:Стандарты разработки V8) — проверки модулей в EDT: транзакции в Попытка–Исключение, документирующие комментарии, структура модуля и регионов, имена переменных и параметров, запросы не в цикле, безопасный режим, удаление временных файлов.
- **BSL Language Server** — диагностики по стандарту, SQL, производительности, сложности, безопасности, устаревшим методам; каноническое написание ключевых слов, алиасы в запросах, описание параметров и возвращаемых значений.

Подробности и ссылки — в STANDARDS-AND-TOOLS.md.
