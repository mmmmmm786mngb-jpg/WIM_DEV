Файл: DataProcessors/ЭкспортПорученийВЛуч/Ext/ManagerModule.bsl, стр. 129-132

Проблемный код

// Строка 129: условие отбора по периоду
|   И СостоянияДокументовВФайловомШлюзе...Период МЕЖДУ &ЧасНазад И &ДеньНазад";

// Строки 130-132: установка параметров

Запрос.УстановитьПараметр("ЧасНазад",  ТекущаяДата - СекундВЧасе());   // например: 16:00
Запрос.УстановитьПараметр("ДеньНазад", ТекущаяДата - СекундВСутках()); // например: вчера 17:00

Суть проблемы

Оператор МЕЖДУ A И B в 1С (как и в SQL) требует A <= B. Здесь:

  • &ЧасНазад = ТекущаяДата - 3600 сек (например, 17.03.2026 16:00)
  • &ДеньНазад = ТекущаяДата - 86400 сек (например, 16.03.2026 17:00)

Левая граница (ЧасНазад) больше правой (ДеньНазад), поэтому интервал пуст. Запрос никогда не вернет строки.

Визуализация проблемы

Временная ось:
  |-------- ДеньНазад (16.03 17:00) ---- ... ---- ЧасНазад (17.03 16:00) ---- Сейчас (17.03 17:00) ----|
                ^                                        ^
             Правая граница                      Левая граница
             (раньше)                            (позже)
  
  МЕЖДУ [позже] И [раньше] ==> пустой диапазон!
Воздействие: процедура контроля задержки отправки никогда не обнаружит зависших пакетов. Сотрудники не получат уведомлений о том, что шлюз не обрабатывает исходящие пакеты.

Рекомендация по исправлению

// Поменять параметры местами:
|   И ...Период МЕЖДУ &ДеньНазад И &ЧасНазад";
// Теперь: "от (сутки назад) до (час назад)" - корректный интервал

    КонецЕсли;
КонецЕсли;

Анализ

Код использует унифицированный подход: проверяет СтатусКвитанции и ДоставленоУспешно, вычисленные из тегов NDC_CHECKING / RECIPIENT_CHECKING внутри SERVICE_MARKS. Этот подход функционально покрывает сценарии TRNOC и TRNRC, так как:

  • TRNOC (уведомление о доставке): приходит как квитанция с STATE="NDC_TO_ORIG" и содержит результаты проверки NDC. Если NDC_CHECKING.RESULT="SUCCESS", то ДоставленоУспешно еще не будет Истина (нужен и RECIPIENT_CHECKING), поэтому статус не изменится.
  • TRNRC (квитанция о приеме): приходит с STATE="RECIP_TO_NDC" и содержит RECIPIENT_CHECKING. Только когда оба статуса SUCCESS, ДоставленоУспешно = Истина.

Однако, согласно ТЗ, при успешном TRNOC должен устанавливаться промежуточный статус ВыгруженоВебКабинет (подтверждение, что НРД принял пакет и передал контрагенту). Этого нет.

Также отсутствует явная обработка BTRAN (batch transfer notification), упомянутого в ТЗ.

Воздействие: промежуточный статус ВыгруженоВебКабинет фиксируется только при обнаружении каталога пакета в зоне "отправленных" (ОбработатьОтправленныеПакеты), но не из квитанций. Это может привести к несогласованности статусов. Отсутствие обработки BTRAN может означать пропуск финальных подтверждений по части сценариев.
