Введение
Сделать механизм, с помощью которого будут автоматически формироваться выписки о состоянии/операциях по счету депо клиентов, с последующей отправкой выписки клиенту через интеграционный шлюз (ИШ). Предполагается, что клиент подаст одно поручение на отправку выписок за каждый день с даты поручения. И нужно будет ежедневно создавать и привязывать выписки к этому поручению, которое будет находится в статусе "Обрабатывается". Когда клиент подаст еще одно поручение на отмену отправки таких выписок, только тогда нужно прекратить формирование и отправку выписок, поручение перевести в статус "Исполнено".

Как есть
Сотрудник депозитария создает документ "Поручение" с видом операции: "Выписка об операциях по счету ДЕПО", "Выписка о состоянии счета ДЕПО", "Отчет об операциях по счету ДЕПО". У документа заполняется "Дата исполнения", документ проводится, статус меняется на "Исполнено".
При смене статуса на "Исполнено" создаются связанные с "Поручением" документы "Выписка" и "Формирование отчета/выписки".
В документе "Выписка" по кнопке "Сформировать" выбирается команда "Выписка по счету депо", после чего формируется печатная форма выписки, которая выгружается в интеграционный шлюз в формате ".pdf".
Как надо
С клиентом подписывается доп. соглашение на ежедневную отправку выписок по счету депо.
Сотрудник депозитария создает документ "Поручение" с видом операции "Выписка о состоянии счета ДЕПО (ежедневно)" (создать такой вид операции) и указывает в нем дату начала отправки выписок.
При записи этого поручения оно перейдет в статус "Обрабатывается".
Когда с клиентом будет подписано доп. соглашение на отмену рассылки этих выписок, тогда сотрудник депозитария заполнит в данном поручении дату исполнения, и поручение перейдет в статус "Исполнено".
Ежедневно (предварительно, в 12:00) будет отрабатывать рег. задание, которое будет искать поручения с видом операции "Выписка о состоянии счета ДЕПО (ежедневно)", у которых статус = "Обрабатывается", и дата начала отправки выписок является рабочим днем и ранее текущей даты. Для каждого из этих поручений за предыдущий день, если он рабочий, будет автоматически сформирована выписка (и формирование отчета/выписки) и её печатная форма (в формате ".pdf") будет выгружена в интеграционный шлюз (для чего нужно будет создать документы "Формирование отчета/выписки", "Выписка", "Техническая опрация" и "Исходящий пакет в файловый шлюз".
Требования на доработку 1С
Доработать справочник "Способы передачи информации"
Добавит предопредленное значение "Интеграционный шлюз (ИШ)".
Доработать документ "Поручение"
Добавить реквизит "Периодичность" с значением типа ПеречислениеСсылка.ПериодичностиРасписанийРассылокОтчетов.
Реквизит "Периодичность" должен быть видим только для поручения с "Видом операции" = "Выписка о состоянии счета ДЕПО", допускается только пустое значение и "Ежедневно".
Следующие требования должны выполняться только для поручения с "Видом операции" = "Выписка о состоянии счета ДЕПО" и переодичностью "Ежедневно":

Реквизит "Номер" документа должен иметь префикс "И".
На форме документа для поля "Уведомление о состоянии счета депо на" (реквизит ДатаНачПериода) выводить заголовок "Уведомление о состоянии счета депо с". Поле "по" (реквизит ДатаКонПериода) скрыть. Переключатель "операционного дня" установить в положение "На конец", сделать запрет на его редактирование.
При записи/проведении документа сделать проверку, что в поле "Уведомление о состоянии счета депо с" заполнен рабочий день, и в поле "Счет депо" заполнен счет, у которого владелец имеет код НРД (справочник "Коды загрузки", реквизит "Биржа" = "NRD code"). Иначе, выводить ошибку и отменять запись/проведение. При проведении документа установить "Статус" = "Обрабатывается".
При проведении документа не создавать по нему документы "Выписка" и "Формирование отчета/выписки".
Доработать рег. задание "Отправка отчета об операциях по счету депо в формате XLS"
Изменить "Имя" и "Синоним" задания на "Отправка ежедневных выписок по счету депо".

Задание должно выполнять следующие действия (ниже описаны требования по созданию обработки "Отправка выписок по счету депо вручную", возможно нужно всё реализовать в обработке и её вызывать в задании):

Проверить, что текущий день рабочий. Если нет, то завершить выполнение задания. Иначе, выполнять требования ниже.
Отобрать проведенные документы "Поручение" с "Видом операции" = "Выписка о состоянии счета ДЕПО" и периодичностю "Ежедневно", у которых "Статус" = "Обрабатывается", и дата в поле "Уведомление о состоянии счета депо с" ранее текущей даты.
Для каждого найденного поручения создать документ "Формирование отчета/выписки" и документ "Выписка" с информацией за предыдущий рабочий день (на конец дня). В целом, эти документы по текущему алгоритму, который срабатывает при проведении поручения, когда в поле "Вид операции" выбрано значение перечисления "Виды отчетов". Если уже есть выписка по этому же счет с информацией на эту же дату, то не создавать новые документы.
В выписке поле "Исх. номер" не заполнять, в поле "Документ выдать" заполнять "Интеграционный шлюз (ИШ)".
Для каждого созданного документа "Выписка" нужно сформировать отчет "Выписка о состоянии счета депо" (вручную создается по команде "Сформировать" -> "Выписка по счету депо" на форме выписки). Форму полученного отчета в формате ".pdf" нужно передать в интеграционный шлюз НРД - создать каталог в каталоге OUTBOX и в нем сохранить этот файл ".pdf". Имя каталога начинается с W, далее идет номер дня в месяце (два символа), далее номер месяца (в шестнадцатеричном формате, 1 символ), далее номер пакета в дне (четыре символа). Имя файла создается по шаблону "Выписка_о_состоянии_счета ДЕПО_N_[Номер счета ДЕПО]_на_[дата]". Также создается файл с фиксированным именем winf.xml (требования по этому файлу см. ниже). Сделать функционал по аналогии с текущим функционалом взаимодействия со шлюзом (дальше данные для информации):
Отправка данных по документу "Раскрытие списка владельцев". Файл winf.xml создается из макета "Исходящий шаблон транзит WINF".
Используются обработки "Файлы НРД" и "Экспорт поручений в Луч".
Для отправки данных создаются документы "Техническая операция" и "Исходящий пакет в файловый шлюз".
Статусы по отправке записываются в регистр сведений "Состояния документов в файловом шлюзе" и в регистр "Документы выгруженные в Луч" .
Для информации, похожий функционал описан в задаче ACCOUNTING-17091 Отправка брокерских отчетов в спец. депозитарии через транзит ИШ
При отмене проведения поручения, отменять его проведение, не удалять связанные с ним документы "Выписка" и "Формирование отчета/выписки".

Требования по заполнению файла
CONTRAGENT_CODE (ORIGINATOR) - выводить код НРД для ЮЛ организации (НРБанк).
CONTRAGENT_CODE (RECIPIENT) - выводить код НРД владельца счета (справочник "Коды загрузки", реквизит "Биржа" = "NRD code"), указанного в поле "Счет депо" поручения.
LETTER_ID - выводить GUID созданного документа "Техническая операция".
CONTR_LETTER_ID - выводить реквизит "Относительный путь" созданного документа "Исходящий пакет в файловый шлюз" (подкаталог с пакетом).
LETTER_TYPE - выводить значение "NDC0STND".
DATE - выводить дату создания документа "Техническая операция".
TIME - выводить время создания документа "Техническая операция".
CONTR_DOC_ID - выводить реквизит "Относительный путь" созданного документа "Исходящий пакет в файловый шлюз" (подкаталог с пакетом).
FILE_NAME - выводить реквизит "Относительный путь" созданного документа "Исходящий пакет в файловый шлюз" (подкаталог с пакетом).
DOC_TYPE - выводить значение "TRZT".
ORIGINAL_FILE_NAME - выводить название файла выписки (PDF) вместе с расширением файла.
Создать обработку "Отправка ежедневных выписок по счету депо вручную"
Нужно, чтобы на форме обработки пользователь смог выбрать список документов "Поручение", указать дату, и при выполнении обработки по этим поручениям за эту дату должны создаваться документы "Выписка" и "Формирование отчета/выписки", и каждую выписку в формате ".pdf" нужно передавать в интеграционный шлюз НРД. То есть нужно реализовать тот же алгоритм, который отрабатывает в рег. задании "Отправка ежедневных выписок по счету депо".
Доработка отчетов
Проверить, что документ "Выписка", созданный при выполнении рег. задания "Отправка ежедневных выписок по счету депо" и обработки "Отправка ежедневных выписок по счету депо вручную", попадает в отчет "Журнал исходящих документов". Если нет, то доработать отчет, что документ попадал.
Проверить, что документ "Формирование отчета/выписки", созданный при выполнении рег. задания "Отправка ежедневных выписок по счету депо" и обработки "Отправка ежедневных выписок по счету депо вручную", попадает в отчет "Журнал операций". Если нет, то доработать отчет, что документ попадал.
Описание настроек
Необходимо создать обработчик для выполнения следующих настроек на продуктивной базе 1С:

Для рег. задания "Отправка выписок по счету депо" снять флаг "Использование", задать расписание: выполнять каждый день, с 12:00:00, 1 раз в день.
Тестирование
Тестирование на стороне 1С
Создать поручение с "Видом операции" = "Выписка о состоянии счета ДЕПО (ежедневно)".
При записи поручения должна срабатывать проверка, что в поле "Уведомление о состоянии счета депо с" заполнен рабочий день, и у счета депо владелец имеет код НРД.
При проведении поручения по нему не должны создаваться документы "Выписка" и "Формирование отчета/выписки".
Проверить, что при выполнении обработки "Отправка ежедневных выписок по счету депо вручную" и рег. задания "Отправка ежедневных выписок по счету депо" по поручению создается документ "Выписка" и "Формирование отчета/выписки".
После выполнения обработки/рег. задания выше выполнить обработку "Заполнение журналов". Затем проверить, что в отчеты "Журнал исходящих документов" и "Журнал операций" попадают созданные документы "Выписка" и "Формирование отчета/выписки".
Интеграционное тестирование
Проверить, что по созданному документу "Выписка" был сформирован отчет и его результат был выложен в интеграционный шлюз - отдельный каталог в каталоге OUTBOX, и в нем есть файл ".pdf" и файл winf.xml. В winf.xml в теге CONTRAGENT_CODE должен быть заполнен код НРД владельца счета депо из поручения.
Проверить на одном и нескольких документах.

Информация по ИШ НРБ - интеграционный шлюз НРБ, он же файловый шлюз ЭДО НРД:

 Шлюз нужно презапускать каждую неделю. Он работает не на правах сервиса, а на правах обычного приложения. Через неделю согласно нашим политикам на сервере сессия принудительно прерывается. Ситмптом: файлы перестаю исчезать из подкаталога AUTBOX.  Перезапускать шлюз умеет Ильдар.
У нас есть два экземпляра тестового шлюза для 1С и BQ:  НРД код для 1С MC0035000000,  Путь к папке для выгрузки из 1С \\corp\Workgroups\NRB\UAT\GateNRD\OUTBOX; НРД код для BQ MC00350000TR, Путь к папке входящих \\corp\Workgroups\NRB\UAT\GateNRD_BQ\INBOX.   Это позволяет отправить пакеты из одного экземпляра ИШ в другой.  Также можно отправить и в тот же самый если использовать код отправителя также и в качестве кода получателя
Каждый исходящий пакет в ИШ размещается в отдельном подкаталоге с воьмизначным именем: Имя каталога начинается с буквы означающей тип пакета (например W - транзитные пакеты, К - поручения) далее идет номер дня в месяце (два символа), далее номер месяца (в шестнадцатеричном формате, 1 символ), далее номер пакета в дне (четыре символа).  Имя основного файла обычно совпадает с именем директории, но для нетепизированных файлов в транзитных пакетах может быть произвольным. ФШ сам переименует файл в соответствии с Правилами ЭДО, если включена настройка «Автоматически переименовывать исходящие файлы в соответствии с Правилами ЭДО». Примечание. Если настройка «Автоматически переименовывать исходящие файлы в соответствии с Правилами ЭДО» выключена, необходимо контролировать, что в имени файла содержатся только символы: '0'..'9', 'A'..'Z', '.', ' ', '_', '-', '(', ')','[', ']', '~','#' и длина имени не превышает 25 символов.

Для того, чтобы контрагент, которому направлен нетипизированный  транзитный документ, увидел его в каком-либо из WEB-кабинетов НРД (веб-кабинет ДКУ) на закладке  «Доп. сервисы», в winf.xml в теге /COVERING_LETTER/SERVICE_MARKS/SUBJECT  нужно указать префикс NTED#.  

Пример минимально допустимого WINF.XML для нетипизированного транзитного документа, который  контрагент сможет увидеть в WEB-кабинете:

<!DOCTYPE COVERING_LETTER>
<COVERING_LETTER PROCESSING_MODE="LIVE" STATE="ORIG_TO_NDC" TRANSIT="YES">
<RECIPIENT>
<CONTRAGENT>
<CONTRAGENT_CODE>REGS</CONTRAGENT_CODE>
</CONTRAGENT>
</RECIPIENT>
<SERVICE_MARKS>
<SUBJECT>NTED#</SUBJECT>
</SERVICE_MARKS>
<DOC>
<DOC_TYPE>TRZT</DOC_TYPE>
<ORIGINAL_FILE_NAME>doc1.docx</ORIGINAL_FILE_NAME>
</DOC>
</COVERING_LETTER>
