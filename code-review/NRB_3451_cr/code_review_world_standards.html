2. Примененные стандарты оценки

  • Functional correctness: код должен корректно реализовывать все ветки бизнес-логики, включая edge cases.
  • Reliability: решение должно быть устойчивым к конкурентному выполнению, частичным сбоям и нестабильному порядку данных.
  • Data integrity: после завершения алгоритма данные должны оставаться в непротиворечивом состоянии.
  • Maintainability: алгоритм должен быть понятным, детерминированным и удобным для дальнейшего сопровождения.
  • Testability: ключевые ветки должны быть легко покрываемы тестами.
  • Static analysis mindset: Sonar-подобный подход к cognitive complexity, magic values, duplication, readability.

3. Findings

CRITICAL Дефект в ветке с пустым СтарыйКод

Новая процедура запускает поиск существующих активных кодов только при условии:

Если ЗначениеЗаполнено(СтарыйКод) Или СтарыйКод = Неопределено Тогда

Если СтарыйКод = "", поиск не выполняется вообще. После этого код может перейти сразу к созданию нового элемента, не закрыв старые активные записи.

Почему это важно: это прямой риск нарушения целостности данных. В справочнике могут остаться несколько активных кодов НРД у одного эмитента.

Инженерная оценка: для production-системы такая ветка неприемлема, потому что дефект возникает на обычном входном значении, а не на экзотическом сценарии.

Рекомендация: поиск активных кодов должен выполняться всегда. Проверка СтарыйКод не должна влиять на сам факт актуализации.

HIGH Отсутствие атомарности при обновлении нескольких записей

Алгоритм выполняет серию изменений по одной записи за раз через ИзменитьКодЗагрузки(...) и затем при необходимости создает новую запись через СоздатьКодЗагрузки(...).

Проблема: если одна из записей успешно закрыта, а следующая операция упала, система останется в промежуточном состоянии.

Почему это важно: по постановке предусмотрены разные режимы запуска загрузки, включая регламентное задание и ручную обработку. Для таких сценариев отсутствие транзакционной границы увеличивает риск частично примененных изменений.

Рекомендация: обернуть актуализацию набора кодов в транзакцию либо использовать согласованный механизм блокировок и единый commit для всей операции.

HIGH Нет защиты от конкурентного выполнения

Запрос активных кодов не использует явную блокировку набора данных, а изменение выполняется объектно после получения выборки.

Риск: два параллельных запуска могут читать один и тот же набор активных кодов и независимо принимать решение о закрытии/создании. Это способно привести к дублям, гонкам и непредсказуемому финальному состоянию.

Почему это важно: "работает в одиночном потоке" не считается достаточным уровнем качества для корпоративной загрузки справочных данных.

Рекомендация: добавить блокировку по владельцу и классификатору на время операции либо гарантировать сериализацию запуска снаружи.

MEDIUM Алгоритм не отражает бизнес-спецификацию напрямую

В ТЗ описан пошаговый детерминированный сценарий: сначала обработать дубли, затем среди оставшихся найти совпадение с новым кодом, затем либо оставить один активный код, либо создать новый.

Реализация сводит все это к одному проходу по выборке.

Проблема: код труднее проверить глазами, сложнее доказать эквивалентность ТЗ и сложнее сопровождать в будущем.

Рекомендация: реализовать алгоритм в явных фазах или хотя бы выделить отдельные внутренние функции по шагам спецификации.

MEDIUM Отсутствует детерминизм при выборе сохраняемой записи из дублей

Код оставляет активной первую встретившуюся подходящую запись. Порядок выборки не зафиксирован явно.

Даже если бизнес допускает "любую" запись: для production-кода лучше иметь предсказуемый критерий выбора, например по дате начала действия, по ссылке или по дате создания.

Рекомендация: добавить явное правило выбора и закрепить его в коде и комментарии.

MEDIUM Нормализация кода выполняется только при сравнении, но не перед сохранением

Сравнение идет через СокрЛП(Выборка.Код) = СокрЛП(НовыйКод), но новый код сохраняется как есть.

Риск: возможны записи, отличающиеся только пробелами по краям. Это ухудшает качество справочника и может усложнить последующую идентификацию.

Рекомендация: нормализовать НовыйКод один раз в начале процедуры и дальше использовать только нормализованное значение.

MEDIUM Недостаточная тестопригодность и отсутствие доказательства покрытия edge cases

Изменение затрагивает stateful business logic с датами действия, дублями и разными точками запуска. Это класс задач, где ручной просмотр кода недостаточен.

Минимально необходимые сценарии:

  • нет активных кодов;
  • один активный совпадающий код;
  • один активный несовпадающий код;
  • несколько активных кодов, среди них дубли;
  • несколько активных кодов, совпадения нет;
  • пустой СтарыйКод;
  • одновременный запуск двух обработок.

Рекомендация: без такого набора тестов изменение не должно считаться полностью завершенным.

LOW Магические значения пустой даты и их непоследовательное использование

В коде используются ДАТА(1, 1, 1) и ДАТАВРЕМЯ(1, 1, 1) напрямую.

Проблема: это снижает читаемость и провоцирует появление нескольких вариантов одного и того же правила в разных местах.

Рекомендация: вынести "пустую дату действия" в общую функцию или единый helper.

LOW Документирующие комментарии улучшены, но остаются неточности

Вынесение СоздатьКодЗагрузки и ИзменитьКодЗагрузки в менеджер каталога является плюсом. Однако по diff видно, что комментарий к ИзменитьКодЗагрузки начинается как к процедуре создания нового элемента.

Рекомендация: исправить комментарии, чтобы документация не вводила в заблуждение.

4. Что сделано хорошо

  • Исправлен главный функциональный недостаток оригинала: обработка теперь запускается при любом заполненном issuer_nrd, а не только при отсутствии текущего кода.
  • Сужен scope изменений: доработаны только релевантные участки RuDataAPI и менеджера каталога.
  • Запросы параметризованы, нет небезопасной конкатенации.
  • Создание и изменение элементов вынесены в менеджер справочника, что лучше прямого дублирования объектной записи в разных местах.
  • Есть попытка сделать алгоритм идемпотентным: совпавший код остается активным, лишние закрываются.

7. Итоговый вердикт

Вердикт: решение нельзя считать полностью готовым по мировым engineering standards без дополнительной доработки.

С точки зрения бизнеса изменение уже содержит правильную идею и исправляет базовое ограничение оригинала. Но с точки зрения надежности production-кода остаются критичные и high-risk вопросы: ветка с пустым СтарыйКод, отсутствие атомарности, отсутствие защиты от гонок и недостаточная доказуемость алгоритма на дублях.

Рекомендуемый статус: нужна доработка перед финальной приемкой.
