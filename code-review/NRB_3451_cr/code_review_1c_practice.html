Задача: RuData: актуализация кода НРД эмитента при смене issuer_nrd
База: C:\1c\Cursor_1c\WORK\NRB_Depo
Источники: постановка, ОтчетОСравнении.txt, оригинал в NRB_Depo

Как обычно ревьюят BSL в 1С (ориентиры)

В русскоязычной практике ревью часто опираются на три опоры:

  • Чек-лист по разделам (комментарии, оформление, имена, архитектура, дубли, проверки данных, запросы) - типичная структура материалов для code review, см. например статью на Infostart.
  • Система стандартов разработки 1С (ИТС, публичные выжимки вроде v8std.ru): оформление модулей, запросы, транзакции, общие модули.
  • Статический анализ: BSL Language Server, правила SonarQube 1C (BSL) Plugin, EDT v8-code-style.
Ниже - ревью именно в таком духе: сначала быстрый вердикт, затем проход по чек-листу, затем что бы подсветил линтер, затем итог для внедренца.

Краткий вердикт (как на созвоне с разработчиком)

Суть: направление верное, баг оригинала с условием КодЗагрузкиНДЦ закрывается. Но перед мержем в основную ветку нужно поправить ветку с пустой строкой СтарыйКод и прогнать EDT/BSL LS - иначе риск "грязных" активных кодов в справочнике.

Что бы подсветил BSL Language Server / Sonar (типовые категории)

Точный список срабатываний без прогона инструмента на ветке разработчика назвать нельзя. По опыту ревью на таком коде обычно смотрят:

  • Cognitive complexity процедуры АктуализироватьКодЗагрузки - возможно превышение порога, рекомендация разбить на подпроцедуры.
  • Magic numbers / даты - повторяющиеся ДАТА(1,1,1), ДАТАВРЕМЯ(1,1,1) - вынести в общую функцию или константу модуля.
  • Описание параметров в документирующем комментарии - проверка полноты типов и смысла СтарыйКод (зачем параметр, если по факту не фильтрует выборку).
Рекомендация команде: прогнать ветку через EDT (v8-code-style) и BSL LS / Sonar по регламенту проекта и приложить отчет к задаче - это стандарт де-факто для зрелых 1С-команд.

Замечания по существу доработки (по-1С-овски, без "международной теории")

  1. Блокер: исправить условие входа в блок поиска активных кодов, чтобы не пропускать пустую строку СтарыйКод.
  2. Важно: явно зафиксировать порядок обработки дублей (сортировка в запросе или правило "какой элемент оставляем") - иначе при смене СУБД/плана поведение может "плавать".
  3. Желательно: НовыйКод нормализовать один раз (СокрЛП) до сравнения и до записи, чтобы в справочнике не копились варианты с пробелами.
  4. Регресс: прогнать загрузку регламентом и обработкой, плюс сценарии из ТЗ (новые/обновленные эмитенты и прочие признаки загрузчика).
