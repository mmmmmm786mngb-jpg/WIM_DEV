Универсальная настройка отбора по полям договора ДУ (СКД)

Цель для IMAPPS-36274: пользователь на форме настройки нажимает кнопку, открывается компоновщик настроек схемы компоновки данных, задает отборы по доступным полям (в т.ч. по реквизитам, связанным с договором ДУ / портфелем), настройки сохраняются; при регламентном запуске обработка восстанавливает настройки и получает список портфелей.

1. Общая архитектура решения

  1. Макет СКД в конфигурации (например, у справочника Портфели или у общего модуля / обработки): набор данных — запрос к Справочник.Портфели (и при необходимости соединения), в наборе перечислены поля, по которым разрешен отбор.
  2. Параметры схемы (например, ДатаСреза) — для условий «действует на дату», аналогично существующей логике ДействующиеПортфели.
  3. Хранение выбранных пользователем настроек в элементе справочника «Настройки закрытия…»: реквизит типа ХранилищеЗначения или строка/XML с сериализованными настройками.
  4. Сервер: загрузить схему, создать КомпоновщикНастроекКомпоновкиДанных, инициализировать источник доступных настроек, восстановить сохраненные настройки, установить параметры, выполнить компоновку с выводом в коллекцию значений, взять колонку Ссылка (или идентификатор портфеля).
В конфигурации уже есть близкий пример: Справочники.Портфели.СписокПортфелейПоДинамическомуСписку — перенос отбора из динамического списка в настройки СКД макета ДействующиеПортфели и выполнение в таблицу значений. Для настройки расписания вместо динамического списка используется сохраненный снимок настроек компоновщика.

2. Открытие компоновщика на управляемой форме

Типовые приемы платформы 8.3:

  • реквизит формы с типом КомпоновщикНастроекКомпоновкиДанных и привязка к полю формы с видом «Настройки компоновки данных» либо открытие общей формы настройки отчета через передачу адреса схемы во временном хранилище;
  • на сервере: поместить двоичные данные макета СКД или схему в временное хранилище, на клиенте открыть форму с параметром АдресСхемыКомпоновкиДанных / настройками компоновщика (в зависимости от выбранного шаблона формы в конкретной конфигурации).

Обсуждения и примеры для управляемых форм: форум infostart, mista, статьи по «компоновщик настроек в управляемых формах».

3. Сохранение и восстановление настроек

Варианты, распространенные в практике 1С:

  • Сериализация настроек объекта НастройкиКомпоновкиДанных через XDTO или внутреннее представление (в т.ч. обмен через XML для хранения в строке или в ХранилищеЗначения).
  • ЗначениеВСтрокуВнутр / ЗначениеИзСтрокиВнутр для отдельных структур (осторожно с версиями платформы и объемом данных).
  • Отдельный регистр сведений «Сохраненные настройки» или подсистема БСП с хранилищами настроек — если нужны права, версионность и список вариантов.

Примеры и разбор (внешние материалы):

  • 1С:ИТС — глава про систему компоновки данных (руководство разработчика)
  • Заметки по 1С: сохранение отборов СКД
  • forum.mista.ru — компоновщик настроек в управляемых формах
  • forum.mista.ru — сохранение настроек СКД в обработке

5. Риски и проектные решения

  • Совместимость схемы: при изменении запроса СКД старые сохраненные настройки могут стать невалидными — нужна версия схемы или миграция.
  • Производительность: компоновка должна возвращать только поле ссылки и минимально необходимые данные; тяжелые соединения — в запросе набора данных с параметрами.
  • Безопасность: ограничить доступные поля в схеме (не отдавать в отбор лишние таблицы).
  • Сосуществование с заплаткой: приоритет правил: явный список портфелей, затем режим СКД, затем типы/стратегии/пулы — зафиксировать в ТЗ, чтобы не дублировать отборы.

Документ носит обзорный характер; конкретные имена форм и вызовы БСП нужно согласовать с версией конфигурации и наличием типовых форм настройки СКД в WIM_FIn.
