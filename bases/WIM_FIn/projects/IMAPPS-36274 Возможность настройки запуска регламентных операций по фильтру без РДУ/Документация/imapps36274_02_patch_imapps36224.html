Временная доработка IMAPPS-36224 (заплатка по пулам)

Описание основано на комментариях в задаче IMAPPS-36224 и связи с IMAPPS-36274 (системное решение по гибкому отбору планируется отдельно).

Важно про выгрузку в репозиторий: в проанализированной копии исходников WIM_FIn реквизит ВариантФильтраПул и табличная часть Пулы в XML метаданных могут отсутствовать до обновления ветки или синхронизации с коммитом внедрения (в Jira указан MR 255 для ФИН). Ниже зафиксировано целевое поведение по постановке, а не обязательно текущее состояние каждой рабочей копии.

1. Бизнес-цель

Разделить регламентное групповое закрытие периода на потоки с разным составом портфелей по признаку, связанному с розничным ДУ (РДУ), без ручного перечисления договоров. Проблема: тип клиента «Физические лица» не отделяет РДУ от остальных договоров этого типа.

2. Изменения в ФИН (по описанию разработчика)

ОбъектЧто добавлено
Справочник «Настройки закрытия портфелей по расписанию»Реквизит ВариантФильтраПул; табличная часть Пулы (ссылки на пулы)
Обработка «Групповое выполнение закрытия периодов»Те же реквизит и ТЧ для переноса настроек при запуске из регламента

Дополнение логики получения списка действующих портфелей: если выбран один из вариантов фильтра по пулам (включая или исключая указанные пулы), учитывается содержимое ТЧ Пулы (в сочетании с уже существующими отборами по типам/стратегиям/списку).

3. Изменения в обмене ДУ — ФИН

В обработку обмена добавлена выгрузка реквизита Пул для портфелей из ДУ в ФИН, чтобы в ФИН можно было строить отбор по заполненному пулу (пример из теста: исключение пула «Розничное ДУ»).

4. Пример эксплуатации (из Jira)

Настройка: регламентные операции по всем портфелям, кроме тех, у которых указан пул «Розничное ДУ». Позволяет вынести тяжелое РДУ-закрытие в отдельное окно расписания (в обсуждении — второй поток, например 19:00).

5. Почему это «заплатка»

  • Фиксированная предметная модель: отбор завязан на справочник Пул и заранее добавленные реквизиты, а не на произвольные поля договора.
  • Для каждого нового критерия (другой реквизит из ДУ 1.5) снова потребуются доработки метаданных и обмена, если не выносить всё в универсальный механизм (см. IMAPPS-36274 и документ про СКД).
  • В комментарии Ikonnikov K.: системное решение делается в IMAPPS-36274.
