h3. Контекст и бизнес-проблема

* *Что болит:* при большом количестве договоров РДУ долго отрабатывает процесс *"Групповое выполнение закрытия периода"* (раздел закрытия периода).
* *Почему текущих настроек мало:* отбор по типам клиентов и ручному перечню договоров/портфелей неудобен; РДУ часто попадает в тот же тип, что и прочие договоры (например, физические лица), поэтому *исключить РДУ без ручного перечисления* практически невозможно.
* *Связь с временным решением:* ранее использовалась заплатка по задаче IMAPPS-36224; требуется *системное* решение с настройкой в интерфейсе (см. комментарий в задаче).

{info}*Цель для бизнеса:* дать настраиваемый, повторяемый способ сузить состав портфелей для регламентного закрытия *без ежедневного ручного выбора*, с сохранением привычного сценария работы.{info}

----

h3. Что должно получиться у пользователя (результат в двух предложениях)

# В карточке настройки расписания закрытия пользователь задает *дополнительный отбор портфелей* (гибкие условия по доступным полям).
# При запуске группового закрытия (вручную и по расписанию) система сначала формирует список портфелей *как сейчас*, затем *дополнительно* применяет сохраненный отбор. Если отбор не задан, поведение *полностью как до доработки*.

----

h3. Область изменений (понятными терминами)

|| Объект в системе || Что меняется для пользователя / процесса ||
| Настройки закрытия портфелей по расписанию | Появляется возможность сохранить "расширенный фильтр" по портфелям вместе с элементом настройки |
| Групповое выполнение закрытия периодов | При расчете состава портфелей учитывается сохраненный фильтр из выбранной настройки |
| Справочник портфелей (техническая часть) | Добавляется механизм применения сохраненного фильтра к уже отобранному списку (без замены старой логики отбора) |

{note}*Важно для согласования:* доработка *не отменяет* существующие способы отбора (типы, стратегии, списки). Новый фильтр работает *поверх* уже сформированного списка кандидатов.{note}

h3. Пользовательский сценарий (happy path)

# Специалист открывает элемент *"Настройки закрытия портфелей по расписанию"* и настраивает дополнительный отбор портфелей (условия в интерфейсе настройки СКД).
# Сохраняет элемент настройки.
# Запускает (или дожидается регламентного запуска) *"Групповое выполнение закрытия периода"* с этой настройкой.
# В результате в обработку попадает *суженный* набор портфелей в соответствии с отбором.

----

h3. Правила работы фильтра (бизнес-логика)

* *Если фильтр не заполнен или не используется:* система ведет себя *как сейчас*, без изменений для пользователя.
* *Если фильтр заполнен:* к списку портфелей, полученному штатным механизмом, применяется дополнительное сужение.
* *Регламент и интерактив:* одинаковые правила применения фильтра (настройка хранится в элементе справочника и доступна при фоновом запуске).

----

h3. Критерии приемки (проверка бизнесом)

# *Сохранение:* после записи настройки и повторного открытия те же условия отбора отображаются и не теряются.
# *Эффект отбора:* при включенном фильтре итоговый состав портфелей в групповом закрытии соответствует условиям (визуально/по контрольной выборке).
# *Отсутствие регресса:* при пустом фильтре сценарий закрытия не отличается от текущего поведения.
# *Расписание:* при выполнении по расписанию применяется та же настройка, что задана в элементе справочника.
# *Устойчивость:* типовой сценарий закрытия завершается без аварийного прерывания; при сбоях в настройке ожидается корректная диагностика на стороне ИТ (журнал регистрации), без "падения" сеанса.

----

h3. Вне скоупа (отдельные согласования)

* Прокидывание дополнительных полей из ДУ 1.5 в ФИН (упомянуто в обсуждении задачи) в данный объем *не входит* без отдельной постановки.
* Оптимизация производительности "в целом по РДУ" (см. связанные задачи, например IMAPPS-36275) рассматривается отдельно от настройки отбора.
