# Техническое задание на доработку (по фактической реализации FP)

**Задача:** IMAPPS-36274  
**Наименование:** Возможность настройки запуска регламентных операций по фильтру без РДУ  
**Источник требований:** `Jira.txt`  
**Компонент:** AvancoreFinance  
**База:** WIM_FIn  
**Расширение-референс:** `FP`

## 1. Основание и проблема

По описанию в Jira:

- при большом количестве РДУ-договоров процесс "Групповое выполнение закрытия периода" работает долго;
- отбор только по типам клиентов и ручному списку неудобен;
- РДУ попадает в общий тип "Физические лица", что затрудняет исключение РДУ без сложного ручного выбора.

Цель доработки: добавить гибкий пользовательский отбор портфелей в существующую логику закрытия периода, без слома текущего механизма.

## 2. Функциональная цель

1. Пользователь настраивает отбор портфелей в элементе справочника `НастройкиЗакрытияПортфелейПоРасписанию` через СКД.
2. Настройки отбора сохраняются в реквизите типа `ХранилищеЗначения`.
3. При формировании списка портфелей в обработке `ГрупповоеВыполнениеЗакрытияПериодов` сначала выполняется штатный отбор, затем применяется дополнительный фильтр СКД.
4. Если СКД-настройка отсутствует или пустая, поведение остается прежним.

## 3. Объекты метаданных и изменения

### 3.1. Справочник `НастройкиЗакрытияПортфелейПоРасписанию`

**Изменения в объекте:**

- добавлен реквизит `НастройкиОтбораПортфелейСКД` (тип `ХранилищеЗначения` / `v8:ValueStorage`).

**Изменения в форме элемента:**

- подключены события формы:
  - `OnCreateAtServer` (`callType="After"`) -> `fp_ПриСозданииНаСервереПосле`
  - `BeforeWriteAtServer` (`callType="After"`) -> `fp_ПередЗаписьюНаСервереПосле`
- в модуле формы:
  - инициализация компоновщика по макету `Справочники.Портфели.ПолучитьМакет("МакетОтбораПортфелей")`;
  - загрузка сохраненных настроек (или настроек по умолчанию);
  - сохранение настроек компоновщика в реквизит `НастройкиОтбораПортфелейСКД`.

### 3.2. Справочник `Портфели`

**Изменения в менеджере:**

- добавлена экспортная функция `ПолучитьПортфелиПоОтборуСКД(НастройкаЗакрытияПортфелейПоРасписанию, ИсходныйСписокПортфелей)`;
- добавлена процедура `УстановитьПараметрыСКД(Настройки, ИмяПараметра, Значение)`.

**Логика функции `ПолучитьПортфелиПоОтборуСКД`:**

- если нет настройки или исходного списка, возвращается исходный результат;
- берутся настройки СКД из `НастройкаЗакрытияПортфелейПоРасписанию.НастройкиОтбораПортфелейСКД`;
- выполняется макет СКД `МакетОтбораПортфелей`;
- в параметр СКД `СписокПортфелей` передается исходный список;
- результат возвращается как массив ссылок (`ВыгрузитьКолонку("Ссылка")`).

**Изменения в макете СКД:**

- добавлен/используется макет `МакетОтбораПортфелей`;
- используется параметр `СписокПортфелей` в запросе СКД для ограничения выборки.

### 3.3. Обработка `ГрупповоеВыполнениеЗакрытияПериодов`

**Изменения в модуле объекта (`FP/.../Ext/ObjectModule.bsl`):**

- `&Вместо("ЗаполнитьРеквизитыПоНастройкеРегулярногоЗакрытия")`
  - добавлен перенос ссылки настройки: `ЭтотОбъект.НастройкаЗакрытияПортфелейПоРасписанию = НастройкаРегулярногоЗакрытия`;
- `&ИзменениеИКонтроль("ЗаполнитьПериодыПортфелейПоГраницам")`
  - после вызова `ДействующиеПортфели(...)` добавлен фильтр:
  `Справочники.Портфели.ПолучитьПортфелиПоОтборуСКД(...)`;
- `&ИзменениеИКонтроль("ЗаполнитьПериодыПортфелейПоВарианту_ДНГ_ДКР")`
  - добавлен аналогичный вызов фильтра;
- `&Вместо("ЗаполнитьСписокПортфелей")`
  - после получения `ДействующиеПортфели` добавлен вызов `ПолучитьПортфелиПоОтборуСКД(...)`.

Итог: фильтрация по СКД встроена во все ключевые места, где формируется набор портфелей для закрытия.

### 3.4. Технически включенные заимствованные объекты

## 4. Функциональные требования к итоговой поставке

1. В форме элемента `НастройкиЗакрытияПортфелейПоРасписанию` пользователь может задать и сохранить отбор портфелей через СКД.
2. Сохраненный отбор корректно восстанавливается при повторном открытии элемента.
3. В обработке `ГрупповоеВыполнениеЗакрытияПериодов` при заполнении списка/периодов портфелей применяется дополнительный фильтр из настройки.
4. При пустой СКД-настройке результат полностью совпадает со штатной логикой без регрессии.
5. Отбор работает как при интерактивном запуске, так и при запуске по расписанию.

## 5. Нефункциональные требования

1. Не изменять базовый алгоритм штатного отбора портфелей, а только дополнять его пост-фильтрацией.
2. Использовать параметры СКД, без небезопасной подстановки строк в текст запроса.
3. Обеспечить обратную совместимость с существующими настройками расписания.
4. Не ухудшать производительность за счет повторных лишних запросов вне контура СКД.

## 7. Критерии приемки

1. **Сохранение настроек:** после записи и повторного открытия элемента справочника отбор СКД сохранен.
2. **Применение фильтра:** при наличии настройки СКД список портфелей в обработке сужается согласно отбору.
3. **Режим без фильтра:** при пустом `НастройкиОтбораПортфелейСКД` результат идентичен старому поведению.
4. **Регламентный запуск:** фильтр применяется при запуске регламентной операции по настройке расписания.
5. **Стабильность:** отсутствуют ошибки выполнения в типовом сценарии закрытия периода.

## 8. Ограничения и допущения

1. Доработка реализуется поверх текущей бизнес-логики и не заменяет ее.
2. Дополнительные поля из ДУ 1.5 (упоминались в комментарии Jira) в текущий объем не включены и требуют отдельного согласования.
3. Документ основан на фактическом составе расширения `FP` и может использоваться как базовое ТЗ для переноса в основную конфигурацию.
