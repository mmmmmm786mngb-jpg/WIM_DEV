Текущая связь блоков: справочник настроек, обработка, регламент

Контекст: конфигурация WIM_FIn (исходники по пути из bases/WIM_FIn/source-path.txt, обычно C:\1c\Cursor_1c\WORK\WIM_FIn\src\cf).

В предметной области ФИН договор доверительного управления в коде и метаданных чаще называется справочником Портфели (в комментариях к менеджеру справочника явно указано «действующие договоры ДУ»).

1. Справочник «Настройки закрытия портфелей по расписанию»

Назначение: хранить параметры запуска группового закрытия периода по расписанию и привязку к отдельному экземпляру регламентного задания.

1.2. Создание и параметры регламентного задания

В модуле объекта справочника при записи элемента:

  • если задания еще нет и включен флаг ЗаданиеВключено, создается регламентное задание по метаданным ГрупповоеВыполнениеЗакрытияПериодаПоНастройкам;
  • в Параметры задания записывается массив из одного элемента: ссылка на текущий элемент справочника;
  • обновляются использование, имя пользователя, наименование задания.

При пометке удаления настройки флаг включения сбрасывается; при удалении элемента задание удаляется.

2. Регламентное задание «Групповое выполнение закрытия периода по настройкам»

В метаданных метод: ОбщийМодуль.ФоновыеЗаданияСервер.ГрупповоеВыполнениеЗакрытияПериодаПортфелей.

Логика обработчика (упрощенно):

ОбработкаЗакрытия = Обработки.ГрупповоеВыполнениеЗакрытияПериодов.Создать();
ОбработкаЗакрытия.ЗаполнитьПериодыПортфелей(НастройкаРегулярногоЗакрытия);
ОбработкаЗакрытия.ВыполнитьРегулярноеРасписаниеЗакрытияПериода(НастройкаРегулярногоЗакрытия);

То есть вся фильтрация портфелей и дат выполняется внутри обработки после копирования полей из элемента справочника.

3. Обработка «Групповое выполнение закрытия периодов»

3.1. Загрузка настроек из справочника

Процедура ЗаполнитьРеквизитыПоНастройкеРегулярногоЗакрытия переносит в объект обработки:

  • ПоВсемДействующимПортфелям, вариант периода, ТЧ типов клиентов и стратегий;
  • список портфелей из ТЧ НастройкиЗакрытияПериода;
  • СоздаватьОбщиеОперации;
  • даты начала/конца периода по смещениям из настройки и варианту ДН/ДК.

3.2. Как формируется список портфелей (договоров ДУ) в обработке группового закрытия

Признак ПоВсемДействующимПортфелям = Истина:

  • вызывается ЗаполнитьСписокПортфелей(ДатаЗакрытия) — см. также раздел 3.4;
  • внутри неё из ТЧ обработки берутся типы клиентов и при непустой ТЧ стратегий строится массив портфелей через СписокПортфелейПоСтратегиям (регистр ИнвестиционныеПрофили);
  • итоговый массив ссылок: Справочники.Портфели.ДействующиеПортфели(ДатаСреза, ТипКлиентаДУ, СписокПортфелейПоСтратегиям) — запрос к Справочник.Портфели по датам действия, пометке удаления, типу клиента ДУ и при необходимости ограничение «портфель в списке по стратегиям»; шаблонные типы исключаются.

При ПоВсемДействующимПортфелям = Ложь состав портфелей задаётся явно: реквизит СписокПортфелей обработки (загружается из ТЧ НастройкиЗакрытияПериода справочника настроек).

3.3. Функция СписокПортфелейПоДинамическомуСписку — где вызывается и что делает

Место определения: менеджер справочника Портфели (Catalogs/Портфели/Ext/ManagerModule.bsl), экспортная функция СписокПортфелейПоДинамическомуСписку(ДатаПериода, ДинамическийСписок).

Алгоритм:

  • загружается макет СКД ДействующиеПортфели;
  • создаётся КомпоновщикНастроекКомпоновкиДанных, инициализируется источником доступных настроек по этой схеме;
  • в настройки подставляются настройки по умолчанию из схемы, параметр данных ДатаПериода заполняется переданной датой;
  • из ДинамическийСписок.КомпоновщикНастроек.ПользовательскиеНастройки переносятся элементы отбора (тип ОтборКомпоновкиДанных) в отбор настроек компоновки;
  • выполняется компоновка с выводом в коллекцию значений; из результата возвращается колонка Ссылка — массив портфелей, удовлетворяющих и запросу схемы, и пользовательскому отбору динамического списка.

Где используется в конфигурации (поиск по коду): только форма элемента справочника РегламентныеПериоды — Catalogs/РегламентныеПериоды/Forms/РегламентныйПериод/Ext/Form/Module.bsl:

  • ОбновитьРеглОперацииНаСервере — получает массив портфелей по отбору динамического списка Портфели на дату периода, затем строит план/факт регламентных операций через РегистрыСведений.НастройкиЗакрытияПериода и заполняет дерево операций на форме;
  • ПолучитьПортфелипоДинамическомуСписку — снова вызывает эту функцию для того же списка и даты, чтобы согласовать состав портфелей с планом операций при интерактивных действиях на форме регламентного периода.
Связь с расписанием закрытия: цепочка «НастройкиЗакрытияПортфелейПоРасписанию → регламент → ГрупповоеВыполнениеЗакрытияПериодов» не вызывает СписокПортфелейПоДинамическомуСписку. Это другой сценарий UI (отбор в динамическом списке на форме регламентного периода), но та же СКД ДействующиеПортфели и тот же приём переноса отбора.

3.4. Где применяются фильтры из настроек расписания и как

Источник — элемент справочника НастройкиЗакрытияПортфелейПоРасписанию. Перенос в объект обработки выполняется в процедуре ЗаполнитьРеквизитыПоНастройкеРегулярногоЗакрытия (модуль объекта обработки): копируются ПоВсемДействующимПортфелям, ТЧ ТипыКлиентов и Стратегии, значения портфелей из ТЧ НастройкиЗакрытияПериода в список значений СписокПортфелей, варианты и смещения дат, флаг СоздаватьОбщиеОперации.

Дальше вызывается ЗаполнитьПериодыПортфелейПоНастройкеРегулярногоЗакрытия, которая по комбинации ВариантДатыНачалаПериода / ВариантДатыОкончанияПериода вызывает одну из процедур:

  • ЗаполнитьПериодыПортфелейПоПериоду — по каждому дню диапазона при «всем действующим» вызывает ЗаполнитьСписокПортфелей(День); иначе обходит текущий СписокПортфелей и добавляет строки в ПериодыПортфелей;
  • ЗаполнитьПериодыПортфелейПоГраницам и ЗаполнитьПериодыПортфелейПоВарианту_ДНГ_ДКР — используют функцию ЗапросПоГраницам (или аналогичную логику с той же схемой отборов) и для каждого дня при «всем действующим» снова получают ДействующиеПортфели с теми же типами и стратегиями.

Вспомогательные функции ВыбранныеТипыКлиентовДУ и ВыбранныеСтратегии возвращают данные ТЧ только если ПоВсемДействующимПортфелям = Истина; иначе отбор по типам/стратегиям в эти запросы не подмешивается.

В ЗапросПоГраницам условия подставляются в текст запроса к регистру ГраницаАктуальностиЗакрытияПериодов заменой заглушек 1=1…4=4:

  • ограничение по явному списку портфелей (когда не «по всем действующим») — Портфель В (&СписокПортфелей);
  • по типам клиентов ДУ — Портфель.ТипКлиентаДУ В (&ТипыКлиентовДУ);
  • по концу периода — ГраницаАктуальности <= &КонецПериода;
  • по стратегиям — Портфель В (&СписокПортфелейПоСтратегиям), где массив считается СписокПортфелейПоСтратегиям по ТЧ стратегий.

После получения выборки по границам для каждой строки дополнительно проверяется попадание портфеля в множество ДействующиеПортфели(День, …), если включён режим «по всем действующим» — то есть фильтры настроек сочетаются с правилами «действует на дату».

3.5. Окончательный список портфелей (что уходит в закрытие)

Промежуточный результат при режиме «по всем действующим» на конкретную дату хранится в реквизите обработки СписокПортфелей (список значений ссылок на портфели) сразу после ЗаполнитьСписокПортфелей.

Итоговая структура для выполнения регламентного сценария — табличная часть обработки ПериодыПортфелей: для каждой пары «день закрытия — портфель» создаётся строка (колонки День, Портфель; затем заполняется РеглПериод в процедуре ЗаполнитьПериодыПортфелей). Именно ПериодыПортфелей далее используется при многопоточном закрытии (очередь, ВыполнитьРегулярноеРасписаниеЗакрытияПериода и обход по строкам для регламентных операций).

Кратко: фильтры из настроек расписания задают, какие ссылки попадут в эти строки; окончательный перечень работ — не один массив, а набор пар (дата, портфель) в ПериодыПортфелей.

Ограничение, на которое указывает IMAPPS-36274: отбор по типу клиента не отделяет РДУ от прочих договоров с тем же типом (например, «Физические лица»). Универсального отбора по произвольным реквизитам договора в цепочке расписания нет — только типы, стратегии, явный список (и после внедрения IMAPPS-36224 — отдельная логика по пулам).
