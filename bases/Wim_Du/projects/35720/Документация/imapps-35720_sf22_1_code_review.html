3. Code review - findings

CRITICALНеверная база для выбора коэффициента SF (нарушение постановки задачи)

В ПроцентПоТаблицеКоэффициентов() коэффициент выбирается по параметру &СуммаСЧА = ПоказателиСправки.П015, где П015 это инвестиционный доход за период, а не СЧА на последний день периода.

При этом в макете показатель П012 имеет смысл Стоимость чистых Активов на последний день Отчетного периода, руб.. Значит, для выбора коэффициента должен использоваться именно П012 (или напрямую значение СЧА на конец периода из регистра), а не П015.

Риск: систематически неверная ставка и неверная сумма вознаграждения.

Рекомендация: передавать в запрос фактическую СЧА на конец периода (например, П012 или РегистрыСведений.РегистрРСА_СЧА.ЗначенияСЧА_РСА(РасчетнаяДатаОкончания, ДоговорДУ).СЧА) с проверенной размерностью.

CRITICALСкрытая регрессия по расчету БРСА и БРСА_Начальная

Что сделано неправильно: в новой версии отключен рабочий блок расчета БРСА, а в структуру возврата подставляются другие показатели: БРСА = П012, БРСА_Начальная = П011.

Почему это ошибка: поля П012/П011 отражают СЧА/РСА на конец и начало периода, но не расчет БРСА по формуле приложения 1а (раздел 4.3). БРСА должен считаться как максимум расчетных баз после учета вложений/изъятий и вычета MF/SF.

Связка с ТЗ (Приложение 1а, раздел 4):

  • 4.1 = 2.1 + 2.3 - 2.4 -> в коде это БРСА_Расчетная = БРСА_Начальная + СуммаВложения - СуммаИзъятия.
  • 4.2 = 2.2 - MF - SF -> в коде это РСАМинусВознаграждения = РСА_Окончание - ВознаграждениеMF - ВознаграждениеSF.
  • 4.3 = max(4.1, 4.2) -> в коде это итоговый БРСА = Макс(...) (в базовой реализации дополнительно с нижней границей 0).

Как должно считаться корректно:

  • БРСА_Начальная = БазоваяРСА (база на начало отчетного периода).
  • БРСА_Расчетная = БРСА_Начальная + СуммаВложения - СуммаИзъятия.
  • РСАМинусВознаграждения = РСА_Окончание - ВознаграждениеMF - ВознаграждениеSF.
  • БРСА = Макс(0, БРСА_Расчетная, РСАМинусВознаграждения) (логика текущей базовой реализации SF 22).

Риск: неверное значение БРСА попадает в документ и далее в движения регистра РегистрБРСА, что приводит к искажению базы следующих периодов.

Варианты исправления:

  • Вариант 1 (рекомендуется): вернуть исходный расчет БРСА из внSFРасчетВознаграждения_22_СЧА, оставить подстановку ставки SF по новой логике.
  • Вариант 2: вынести расчет БРСА в отдельную функцию с теми же входными параметрами и формулой, затем вызывать ее из РезультатРасчета.
  • Вариант 3: оставить флаг РассчитыватьБРСА и при выключенном флаге явно возвращать прошлое значение БРСА из регистра, но не подменять БРСА показателями П011/П012.

5. Итоговая оценка ревью

Вердикт: Требуются исправления до приемки

Обработка в текущем виде содержит критичные отклонения от постановки задачи и потенциальные регрессии в расчетной логике. Рекомендуется исправить критические замечания, затем повторить контрольное сравнение и прогон сценариев.

Минимальный список проверок после исправления

  1. Проверить выбор ставки на 3 граничных сценариях СЧА (ниже 250 млн, между 250 и 500 млн, выше 500 млн).
  2. Проверить, что в выбор коэффициента передается именно СЧА на конец периода, а не П015.
  3. Сверить сумму вознаграждения с эталонным ручным расчетом.
  4. Проверить корректность значений БРСА и БРСА_Начальная в структуре возврата.
  5. Проверить печатную форму на всех областях (Заголовок/Тело/Подвал), включая подписи по коэффициентам.
