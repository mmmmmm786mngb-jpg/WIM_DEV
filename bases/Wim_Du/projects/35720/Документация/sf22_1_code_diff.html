2.6. РезультатРасчета - ключевое изменение расчета процента Изменено

Это главное отличие SF 22.1 от SF 22. Расчет процента вознаграждения перенесен с фиксированной ставки на выбор по таблице коэффициентов.

Переменная процента

SF 22:

ПроцентВознаграждения = УправлениеПараметрамиДоговораДУ.ПолучитьПараметрДоговораДУ(СтруктураПараметров.ДоговорДУ, ПВХ.ВеличинаВознагражденияЗаУспех, СтруктураПараметров.Дата);

SF 22.1:

ПроцентВознаграждения1 = УправлениеПараметрамиДоговораДУ.ПолучитьПараметрДоговораДУ(...);
// ...позже:
ПроцентВознаграждения = ПроцентПоТаблицеКоэффициентов(СтруктураРасчетаПриростРСА, СтруктураПараметровПериода, ПоказателиСправки, ПроцентВознаграждения1);
Переменная переименована в ПроцентВознаграждения1 (значение по умолчанию), а фактический процент определяется функцией ПроцентПоТаблицеКоэффициентов.

Формула расчета

SF 22:

РачетнаяВеличинаВознаграждения = (СтруктураРасчетаПриростРСА.ПриростРСА - СуммаЗаУправление) * (ПроцентВознаграждения / 100);
ИтоговыйРасчет = ?(СтруктураРасчетаПриростРСА.ПриростРСА > 0, Макс(0, РачетнаяВеличинаВознаграждения), 0);

SF 22.1:

РачетнаяВеличинаВознаграждения = ОКР((СтруктураРасчетаПриростРСА.ПриростРСА - СуммаЗаУправление) * ПроцентВознаграждения / 100, 2);
ИтоговыйРасчет = ?(СтруктураРасчетаПриростРСА.ПриростРСА - СуммаЗаУправление > 0, Макс(0, РачетнаяВеличинаВознаграждения), 0);
ИтоговыйРасчет = Окр(ИтоговыйРасчет, 2);
Два отличия в формуле:
  • Округление ОКР(..., 2) применяется сразу при расчете величины вознаграждения.
  • Условие проверки изменено с ПриростРСА > 0 на ПриростРСА - СуммаЗаУправление > 0 - проверяется именно инвестиционный доход за вычетом MF.

Показатель П021Строка

SF 22: Отсутствует.

SF 22.1:

ПоказателиСправки.Вставить("П021Строка", "");
// Заполняется в ПроцентПоТаблицеКоэффициентов - текст шкалы для печатной формы

Блоки закомментированного кода в SF 22.1

В РезультатРасчета SF 22.1 закомментированы блоки расчета среднего СЧА, расходов и показателей П04-П06:

Эти расчеты были связаны с оценкой расходов и дохода от среднего СЧА и не требуются в новой модели расчета.

Алгоритм работы:
  • Таблица коэффициентов загружается из настроек через XMLЗначение + ХранилищеЗначения.Получить()
  • Текущее значение РСА_Окончание из СтруктураРасчетаПриростРСА сравнивается с диапазонами в таблице
  • Если подходящий диапазон не найден и РСА > 0, выводится предупреждение "Не заполнена таблица коэфицентов по договору"
  • Если дохода нет (РСА_Окончание <= 0), выводится "Дохода нет"
  • Дополнительно формируется текстовая строка для печатной формы с описанием всей шкалы ставок

8 ФормаНастроек/Module.bsl Изм ПриСозданииНаСервере: десериализация ТаблицаКоэффициентов при загрузке формы.

5. Вывод: противоречия и риски

Ниже - аналитический вывод по коду SF 22.1 относительно ожидаемого поведения и согласованности с SF 22. Цель - отметить реальные дефекты и зоны неопределенности, а не повторять описание функционала.

5.1. Критично: возможная ошибка выполнения в ПроцентПоТаблицеКоэффициентов

Проблема: переменная ТаблицаКоэфициентовХранилище инициализируется только внутри ветки Если НастройкиАлгоритма.Свойство("ТаблицаКоэффициентов") Тогда, а вызовы ТаблицаКоэфициентовХранилище.Сортировать(...) и цикл по строкам идут после закрытия этой ветки (строки 1298-1320 в ObjectModule.bsl).

Следствие: если в настройках нет ключа ТаблицаКоэффициентов (старые сохраненные настройки, ручная правка, другой путь вызова), при расчете возможна ошибка обращения к неинициализированной переменной.

Рекомендация: перенести формирование П021Строка внутрь ветки наличия таблицы либо инициализировать пустую таблицу и выходить без сортировки.

5.2. Мертвый код: УстановитьПользовательскиеНастройки

Проблема: в процедуре УстановитьПользовательскиеНастройки результат XMLЗначение(...) присваивается локальной переменной ТаблицаКоэффициентов, не реквизиту объекта и не модульной переменной.

Следствие: вызов процедуры не меняет состояние обработки; фактическая загрузка шкалы идет через десериализацию в ПроцентПоТаблицеКоэффициентов и в модуле формы настроек.

Рекомендация: либо удалить процедуру, либо реализовать осмысленное применение настроек (если платформа вызывает ее по контракту).

5.3. Несогласованность формы: СчитатьНастройки vs ПриСозданииНаСервере

Проблема: СчитатьНастройки подгружает только Периодичность, ДатаНачалаРасчета, РассчитыватьБРСА, ТипПериода. Поля ТипДня и ТаблицаКоэффициентов из договора не читаются.

Следствие: после команды "Считать" на форме пользователь может увидеть устаревшие или пустые коэффициенты и тип дня, хотя в хранилище договора они сохранены через ПодготовитьСтруктуруХранилища.

Рекомендация: дополнить СчитатьНастройки теми же ключами, что и ПриСозданииНаСервере (включая десериализацию таблицы).

5.4. Вероятная опечатка в ПриСозданииНаСервере

Код: при незаполненной Объект.Периодичность выполняется Объект.ТипПериода = Перечисления.Периодичность.Квартал.

Риск: смешение смыслов: ТипПериода в метаданных обычно "Календарный / Инвестиционный", а не значение перечисления периодичности. Вероятнее ожидалось Объект.Периодичность = Перечисления.Периодичность.Квартал.

Рекомендация: проверить на тестовой базе и исправить при подтверждении.

5.5. Две копии логики расчета в модуле формы настроек

Проблема: в ФормаНастроек/Module.bsl остается устаревшая копия РезультатРасчета и связанных функций (без ПроцентПоТаблицеКоэффициентов, со старой формулой процента из параметра договора).

Следствие: команда проверки результата на форме может расходиться с фактическим расчетом документа через объектный модуль обработки.

Рекомендация: синхронизировать копию с объектным модулем или вызывать один общий серверный метод без дублирования.

5.6. Семантика ТЗ: СЧА в тексте, РСА_Окончание в запросе

В ТЗ формулировка про ставку от СЧА. В запросе ПроцентПоТаблицеКоэффициентов в параметр попадает СтруктураРасчетаПриростРСА.РСА_Окончание (в актуальной версии ИнвестиционныйДоход_Базовый для окончания берется ЗначенияСЧА_РСА(...).СЧА - имена полей совпадают с регистром).

Это не автоматически противоречие, если в вашей методике СЧА на дату отчета совпадает с используемым показателем. Имеет смысл зафиксировать в регламенте: по какому именно значению (СЧА/РСА, какая дата) выбирается строка шкалы.

5.7. Несколько подходящих диапазонов в таблице

Запрос выбирает строки без ПЕРВЫЕ 1 и без агрегата. При пересечении интервалов [МинГраница, МаксГраница] возможны несколько строк; Выборка.Следующий() возьмет произвольную первую.

Рекомендация: валидация при вводе таблицы (непересекающиеся диапазоны) или доработка запроса (например, выбор по максимальной нижней границе).

5.8. Прочие замечания (низкий приоритет)

  • Версия обработки: SF 22.1 помечен как 1.00, SF 22 - 1.02. Численно меньше, хотя это новая ветка - возможна путаница при сопровождении.
  • Сообщить в серверном коде: в ПроцентПоТаблицеКоэффициентов вызовы Сообщить при фоновом расчете могут быть неочевидны пользователю; предпочтительнее журнал регистрации (как в блоке Исключение РезультатРасчета).
  • Опечатки в строках: "коэфицентов", в макете/текстах возможны "управлнеи" и т.п. - на расчет не влияют, на отчетность влияют.
  • Изменение условия итога SF: переход с ПриростРСА > 0 на ПриростРСА - СуммаЗаУправление > 0 меняет бизнес-смысл относительно SF 22 - это осознанное изменение, а не баг, но его нужно согласовать с ТЗ/бухгалтерией.
  • РасчетСуммыЗаУправление: смена начала периода с квартала на ДатаНачалаРасчета меняет выборку проводок MF относительно SF 22 - требуется сверка с методикой квартального учета.

5.9. Итоговая оценка

Противоречий в смысле "два разных процента одновременно" в одном расчете не выявлено: основной поток единообразно опирается на таблицу коэффициентов.

Реальные проблемы: критичный риск ошибки при отсутствии ТаблицаКоэффициентов в настройках (п. 5.1), мертвая процедура (5.2), рассинхрон формы "Считать" и дублирующий модуль проверки (5.3, 5.5).

Остальное - уточнения методики, контроль качества данных в таблице диапазонов и согласование изменений формул относительно SF 22 и ТЗ.
