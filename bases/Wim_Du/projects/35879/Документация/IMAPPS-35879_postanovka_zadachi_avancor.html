IMAPPS-35879: Постановка задачи вендору Аванкор

Перенос доработок из расширения optOst в базовую конфигурацию. Документ предназначен для передачи вендору базовой конфигурации (Аванкор) с целью включения исправлений в штатную поставку.

1. Зачем это было сделано

В модуле объекта документа Сделка при формировании запросов для таблицы продаж БУ и для НКД используются поля регистра бухгалтерии Хозрасчетный, в которых субконто (Субконто2, Партия) хранят полиморфные ссылки на документы разных типов (Сделка, ПогашениеОблигаций, ДокументРасчетовСКонтрагентом, СделкаРЕПОПерваяНога, ВводВыводЦенныхБумагИзДУ, ВводНачальныхОстатовПоНКД, СделкаРЕПОВтораяНога, ПринятиеКУчетуАктивов, СделкаНаСрочномРынке).

Прямое обращение к свойству Дата по такой ссылке в языке запросов (Субконто2.Дата, Партия.Дата) на полиморфной ссылке может отрабатывать нестабильно в зависимости от версии платформы и особенностей оптимизатора запросов. Это приводило к риску некорректной сортировки/группировки по дате документа оприходования и к неоднозначным результатам выборки.

Цель доработки: сделать получение даты по полиморфной ссылке однозначным и воспроизводимым, явно приводя ссылку к конкретному типу документа через ТИПЗНАЧЕНИЯ и ВЫРАЗИТЬ(... КАК Документ.XXX).

Итог: исправление направлено на корректное извлечение даты документа из полиморфных ссылок в запросах документа Сделка и должно быть включено в базовую конфигурацию, чтобы не зависеть от расширения и обновляться вместе с типовой поставкой.

2. Что было сделано в расширении optOst

В расширении конфигурации optOst выполнены подмены двух функций модуля объекта документа Сделка:

  • ТекстЗапросаТаблицыПоПродажамБУ_БезНКД → optOst_ТекстЗапросаТаблицыПоПродажамБУ_БезНКД
  • ТекстЗапросаНКД → optOst_ТекстЗапросаНКД

Все изменения в коде помечены комментариями // IMAPPS-35879 ++ и // IMAPPS-35879 --.

2.1. Функция ТекстЗапросаТаблицыПоПродажамБУ_БезНКД

Заменены три участка текста запроса:

№ Место в запросе Суть изменения
1 Блок «ОстаткиАктивов», поле ДокументОприходованияДата Вместо ЕСТЬNULL(ВТ_ХозрасчетныйОстатки.Субконто2.Дата, ...) используется выражение ВЫБОР по ТИПЗНАЧЕНИЯ(Субконто2) с приведением к каждому из типов документов и получением .Дата; в остальных случаях — ДАТАВРЕМЯ(1,1,1,0,0,0).
2 Тот же блок, выражение в СГРУППИРОВАТЬ ПО Аналогичное выражение для группировки (та же логика ВЫБОР по типам для даты).
3 Блок с ВТ_ОстаткиТело и ВТ_ОстаткиПереоценка, поле Дата Вместо ЕСТЬNULL(ВТ_ОстаткиТело.Партия.Дата, ВТ_ОстаткиПереоценка.Партия.Дата) используется ВЫБОР по ТИПЗНАЧЕНИЯ(ЕСТЬNULL(ВТ_ОстаткиТело.Партия, ВТ_ОстаткиПереоценка.Партия)) с приведением к каждому типу документа и получением .Дата.

Во всех трёх местах перечисляются одни и те же типы документов: Сделка, ПогашениеОблигаций, ДокументРасчетовСКонтрагентом, СделкаРЕПОПерваяНога, ВводВыводЦенныхБумагИзДУ, ВводНачальныхОстатовПоНКД, СделкаРЕПОВтораяНога, ПринятиеКУчетуАктивов, СделкаНаСрочномРынке.

2.2. Функция ТекстЗапросаНКД

Заменён один участок (при признаке «Вести учет купонного дохода в разрезе партий»):

  • В выборке «ТаблицаНКДКСписанию» поле Дата для ВТ_ОстаткиТело.Партия: вместо прямого ВТ_ОстаткиТело.Партия.Дата используется тот же подход — ВЫБОР по ТИПЗНАЧЕНИЯ(ВТ_ОстаткиТело.Партия) с приведением к перечисленным типам документов и получением .Дата.

3. Что необходимо перенести в базовую конфигурацию

В базовый модуль объекта документа Сделка нужно перенести все фрагменты кода между маркерами // IMAPPS-35879 ++ и // IMAPPS-35879 -- из расширения optOst (файл Расширения/optOst/Documents/Сделка/Ext/ObjectModule.bsl).

Конкретно:

  1. В функции ТекстЗапросаТаблицыПоПродажамБУ_БезНКД:
    • Поле ДокументОприходованияДата в блоке «ОстаткиАктивов» (выражение ВЫБОР по ТИПЗНАЧЕНИЯ(Субконто2) с перечислением типов документов).
    • Выражение в СГРУППИРОВАТЬ ПО этого же блока (то же выражение для даты).
    • Поле Дата в блоке с ВТ_ОстаткиТело и ВТ_ОстаткиПереоценка (ВЫБОР по ТИПЗНАЧЕНИЯ для партии).
  2. В функции ТекстЗапросаНКД:
    • Поле Дата в выборке «ТаблицаНКДКСписанию» (ВЫБОР по ТИПЗНАЧЕНИЯ(ВТ_ОстаткиТело.Партия) с приведением к типам документов).

Маркеры // IMAPPS-35879 ++/-- при переносе в базу можно удалить; в расширении они использовались только для идентификации изменённых фрагментов.

4. Проверка после переноса

  • Убедиться, что во всех указанных участках даты формируются через ВЫБОР ... ТИПЗНАЧЕНИЯ(...) ... ВЫРАЗИТЬ(... КАК Документ.XXX).Дата, без прямого обращения к .Дата по полиморфной ссылке.
  • Сравнить результаты выборки по контрольным сделкам до и после переноса (таблица продаж БУ и списание НКД) и убедиться в совпадении ожидаемых данных и порядка.

5. Справочная информация

  • Задача: IMAPPS-35879.
  • Объект: Документ Сделка, модуль объекта.
  • Функции: ТекстЗапросаТаблицыПоПродажамБУ_БезНКД, ТекстЗапросаНКД.
  • Источник доработок: расширение конфигурации optOst, файл Documents/Сделка/Ext/ObjectModule.bsl.
